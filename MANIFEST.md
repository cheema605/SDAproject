## Requirements Compliance

### Functional Requirements
- ✅ **Lab Management**: Create labs with ID, name, venue, schedule
- ✅ **Staff Assignment**: Assign instructors and multiple TAs per lab
- ✅ **Schedule Management**: Set expected start/end times for labs
- ✅ **TimeSheet Recording**: Attendants enter actual session times
- ✅ **Absence Tracking**: Record leaves (absences) by leaving times empty
- ✅ **Makeup Lab Requests**: Instructors request, attendants approve, record actual times
- ✅ **Contact Hours Calculation**: Automatic computation from session times
- ✅ **Three Required Reports**:
  - Weekly schedule (all labs scheduled for year/week)
  - Weekly timesheet (all actual sessions for year/week)
  - Lab semester report (all sessions with totals and makeup info)

### Technical Requirements
- ✅ **Language**: Java 23
- ✅ **UI**: JavaFX 25
- ✅ **Persistence**: Binary serialization (no DBMS)
- ✅ **File Format**: Binary object serialization to `data/datastore.dat`
- ✅ **Build Tool**: Maven 3.11.0
- ✅ **Code Quality**: SOLID principles throughout

### Architecture Requirements
- ✅ **SOLID Principles**: 
  - Single Responsibility per class
  - Open/Closed for extension
  - Liskov Substitution (Person hierarchy)
  - Interface Segregation (service methods)
  - Dependency Inversion (DataStore abstraction)
- ✅ **Layered Architecture**: Domain → Persistence → Services → Presentation
- ✅ **Design Patterns**: Repository, Service Locator, Observer (implicit), Template Method
- ✅ **Extensibility**: Easy to add new reports, services, or backends

## How to Use (Summary)

### Quick Start

```powershell
mvn clean compile

```

### Main UI Features
| Button | Action |
|--------|--------|
| Load | Load labs from `data/datastore.dat` |
| Save | Save labs to `data/datastore.dat` |
| Add Lab | Create new lab (dialog) |
| Assign Instructor | Assign instructor to selected lab |
| Assign TA | Assign TA to selected lab (multiple) |
| Enter TimeSheet | Record actual session time (dialog) |
| Approve Makeup | Approve pending makeup request (dialog) |
| Request Makeup | Instructor requests makeup (dialog) |
| Schedule Report | Generate weekly schedule (year/week input) |
| Weekly TimeSheet | Generate weekly sessions (year/week input) |
| Lab Semester Report | Generate per-lab semester summary |

### Data Flow
1. User creates labs → added to in-memory `ObservableList<Lab>`
2. User performs actions (assign staff, enter times) → updates lab objects
3. User clicks "Save" → `LabRepository` serializes `DataStore` to `data/datastore.dat`
4. User clicks "Load" → `LabRepository` deserializes from file, updates UI table
5. Reports query in-memory `DataStore` and generate text summaries

## Known Limitations & Notes

### Current Implementation
- Single-user (no authentication/authorization)
- In-memory operation (fast, suitable for small datasets)
- Simple UI (no complex layouts or styling)
- Week calculation based on US locale (Monday start)

### Not Implemented (Out of Scope)
- CSV/PDF export (easily added)
- Multi-user concurrent access
- Database backend (can be added without domain code changes)
- Email notifications
- Web UI
- Mobile app

### Assumptions
- Year/week values entered by user match their test data
- Lab IDs are unique
- Instructor/TA names are sufficient for tracking (no complex hierarchy)
- All data fits in memory (not a concern for typical semester size)
